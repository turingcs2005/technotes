Technical documentation

1. Purpose of technical documentation

  • Cross-training: learn an app built by a colleague and service (fix bugs, enhance) the app during his/her absence.
  • Reference: you may forget the details of your own app after a few months.

2. What to include in a tech documentation?

We need to strike a balance between documentation complexity and usefulness. Our apps are constantly evolving with many enhancements being added regularly over time. Maintaining detailed documentation will cost too much: as an app evolves, detailed documentation quickly becomes obsolete and needs update.

Include the following in your tech documentation:

  • Main functions: what does the app do?
  • Workflow: when a user accesses the app, what are the steps to get his/her job done?
  • Data flow: how data are loaded from a database or saved to a database from the front end?
  • Data structure: data object hierarchies. For example:
    • Under a policy there is an array of coverages.
    • Under property coverage there is an array of buildings.
  • Complex business logics/math formulas.
